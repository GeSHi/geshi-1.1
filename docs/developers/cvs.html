Commit Guidelines

  • Document the changes, the changes should be a short and broad summary of the diff
  • document why you made your changes, if appropriate. A diff can't tell us that unless it's a parse error
  • never have an empty comment at all
  • your commits go to a mailing list and to other developers, so don't swear
  • English only
  • Mention bug numbers (at or near the front of the line) if you are fixing or partially fixing a bug, or perhaps if you've regressed a bug
  • Keep it reasonably short (I would expect most commits to be 50-300 chars)

examples

"Optimised method foo" is NOT good: to vague

"Optimised method foo by applying algorithm bar, which means that method baz doesn't need to be called so often" is good: contains a what (optimised method foo), a how (applied algorithm bar, if someone is interested they can read the diff), and a why.
